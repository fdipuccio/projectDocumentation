MODULE: PM VERSION: 1
## 1. Obiettivo
Sviluppare un backend per un sistema di autenticazione utenti che consenta la registrazione, login con email e password, gestione di token JWT, protezione di endpoint e una funzionalità base di reset password, utilizzando lo stack tecnologico indicato.

## 2. Contesto e vincoli
- Il backend deve essere sviluppato utilizzando FastAPI come framework.
- Il database relazionale di riferimento è PostgreSQL.
- L'autenticazione utilizzerà JSON Web Tokens (JWT) per la gestione delle sessioni e protezione degli endpoint.
- Il sistema deve garantire il funzionamento minimo con registrazione, login, reset password e protezione endpoint.
- Le funzionalità di reset password devono essere implementate in modalità "base", senza specifiche avanzate come 2FA o invio email sofisticato (a meno che non venga specificato diversamente).
- Tecnologia principale: Python, FastAPI, PostgreSQL, JWT.

## 3. Assunzioni
- Gli utenti si registrano utilizzando email e password.
- L’email è considerata un identificatore univoco.
- Il reset password base comprende la generazione di un token di reset e modifica della password, senza necessariamente integrare invio email o captcha (o tali meccanismi saranno integrati successivamente).
- La validazione delle password segue criteri base (es. lunghezza minima).
- Il sistema si concentrerà su API RESTful.
- I token JWT includono claims standard sufficienti per l’autenticazione senza estensioni personalizzate.
- L’ambiente di deployment e sicurezza (es. HTTPS, gestione chiavi private JWT) è gestito a livello infrastrutturale e non rientra nello scope.

## 4. Scope MVP
- Implementazione endpoint REST per:
  - Registrazione utente (creazione nuovo record utente con hash della password).
  - Login tramite email e password (verifica credenziali e rilascio token JWT).
  - Generazione e validazione JWT.
  - Endpoint protetti da meccanismo di autenticazione basato su JWT.
  - Reset password base: generazione token reset e modifica password.
- Persistenza dati utenti su database PostgreSQL.
- Documentazione API base (es. OpenAPI autogenerated da FastAPI).

## 5. Out of scope
- Funzionalità avanzate di reset password (invio email, link di reset cliccabili).
- Autenticazione multi-fattore (MFA).
- Gestione ruoli e permessi avanzati.
- Frontend o interfacce utente.
- Integrazione con provider esterni di identità (OAuth, SSO).
- Monitoraggio, logging esteso e auditing.
- Scalabilità orizzontale o performance tuning specifico.

## 6. Task tecnici ordinati
1. Configurare ambiente FastAPI con connessione a PostgreSQL.
2. Definire modello dati utenti e migrazioni per PostgreSQL.
3. Implementare endpoint di registrazione:
    - Validazione input.
    - Hashing sicuro della password (ad es. bcrypt).
    - Salvataggio dati su DB.
4. Implementare endpoint di login:
    - Verifica credenziali.
    - Rilascio token JWT con scadenza.
5. Configurare middleware o dipendenza FastAPI per protezione endpoint tramite JWT.
6. Implementare endpoint protetti di esempio per verifica accesso autenticato.
7. Implementare funzionalità reset password "base":
    - Generazione token reset associato a utente.
    - Endpoint per modifica password con validazione token.
8. Testare integrati e di sicurezza base (es. verifica hashing, validazione token).
9. Documentare API generata in OpenAPI.

## 7. Acceptance criteria
- Gli utenti possono registrarsi creando un utente nel database con password salvata in forma hashata.
- La login con email e password restituisce token JWT valido.
- Gli endpoint protetti sono accessibili solo con token JWT valido; in caso contrario restituiscono errore di autenticazione.
- La funzionalità di reset password consente la generazione di token reset e la modifica della password con token valido.
- Tutte le richieste e risposte sono conformi ai parametri e vincoli FastAPI/PostgreSQL specificati.
- Le migrazioni e modelli dati sono funzionanti e coerenti.
- Documentazione API è disponibile e aggiornata per tutte le funzionalità implementate.
- Nessun requisito extra o ambiguità non risolta rimane da chiarire.

## 8. Rischi e punti aperti
- Ambiguità sulla modalità esatta di reset password (ad esempio, se è richiesta integrazione con invio email o solo token generazione interna).
- Assenza di requisiti relativi a sicurezza avanzata (es. gestione chiavi JWT, revoca token).
- Mancanza di specifiche su metrics di performance o logging può influenzare qualità di produzione.
- Non è specificata la durata o i claims dettagliati del token JWT.
- Il meccanismo di protezione endpoint è supposto basato su JWT standard senza dettagli custom.
- La gestione degli errori e messaggi per il client non è definita, va concordata successivamente.
